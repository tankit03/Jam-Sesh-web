Project Handoff

  • Due Jun 1 by 11:59pm
  • Points 30
  • Submitting a website url or a file upload
  • File Types pdf and zip

This assignment represents the final handover of your project deliverables to your partner organization. It requires formal verification from your project partner that all components meet their requirements and can be independently deployed and maintained.

⚠️ This assignment requires partner verification and sign-off. Plan accordingly as this process may take several days.

If you don't have an external project partner:

  • Have your mentor or instructor serve as the "client" for verification
  • Document how your project could be maintained by future developers

⚠️ If you have deployed your solution on OSU-sponsored hardware or compute resources, they will need to be migrated. All OSU-sponsored compute resources will be shut-down by end of finals' week. Similarly, all hardware purchases made by OSU needs to be returned to instructors before the end of finals' week. Grades will be withheld until this is done.

Deliverables Checklist

Required Documentation

  • Comprehensive README.md with:
    • Project overview and purpose
    • Complete setup instructions
    • Deployment procedures
    • Maintenance guidelines
    • Troubleshooting section
  • Technical documentation (developer-focused)
  • User manual/documentation (if applicable)
  • API documentation (if applicable)
  • Database schema and data dictionary (if applicable)
  • Architecture diagrams
  • List of features or use cases implemented (wrt requirements)
  • Known issues and future enhancements list

Code and Assets

  • Clean, well-commented source code
  • Build/deployment scripts
  • Configuration files (with sensitive information removed)
  • Test cases and test data
  • Any required third-party libraries or dependencies (clearly documented)

Access and Credentials

  • Admin credentials for deployed application
  • Repository access (ensure partner has appropriate permissions)
  • Cloud/hosting service access (if applicable)
  • Database access (if applicable)

Documentation Format Guidelines

  • Store all documentation in your code repository (preferred)
  • Use markdown format for maximum accessibility and version control
  • Avoid proprietary formats where possible
  • Include both source files and rendered versions (e.g., PDF) as appropriate

It's OK to export your documentation from Google Docs or similar to .docx and .pdf (both).

Partner Verification Process

  1. Schedule a formal handover meeting with your partner
  2. Demonstrate your solution working in their environment
  3. Walk through all documentation
  4. Have them attempt to deploy the solution independently using only your documentation
  5. Obtain written verification that they can:
    • Access all code and documentation
    • Build and deploy the solution
    • Administer the deployed application
    • Understand future maintenance requirements

Submission Requirements

  1. ZIP file containing:
    • All source code and assets
    • Complete documentation suite
    • Partner verification document (signed)
    • Deployment evidence, if applicable (screenshots)
  2. For large repositories:
    • Submit a secure Box link to your complete project
    • Ensure link permissions are properly set
  3. Include:
    • Link to public repository (if applicable)
    • Evidence of open-source license compliance (if open-source)

Open Source Considerations

If you're open-sourcing your project:

  • Ensure proper licensing is in place
  • Create an engaging project landing page
  • Include contribution guidelines
  • Set up issue templates
  • Document the project's roadmap
